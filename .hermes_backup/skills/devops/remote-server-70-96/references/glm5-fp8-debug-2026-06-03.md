### Bug 1: Startup Script Sources Wrong venv

`/data/model_startup_script/start_glm5_fp8_2node.sh` line 40 was sourcing the wrong venv:

```bash
# WRONG â€” vllm-latest-cu130 (v0.22.0, no ds4-sm120 source):
source /data/venvs/vllm-latest-cu130/bin/activate

# CORRECT â€” vllm-ds4 (v0.6.0.dev0+cu132, ds4-sm120 editable):
source /data/venvs/vllm-ds4/bin/activate
```

The ds4-sm120 source at `/data/vllm-ds4-sm120/vllm/` is only in the editable install
for vllm-ds4 venv. When vllm-latest-cu130 is active, `import vllm` points to the
pip-installed vllm package (v0.22.0), NOT the git source.

**How to detect:** `bash /data/model_startup_script/start_glm5_fp8_2node.sh 2>&1 | head -3`
If it shows `0.22.0`, the wrong venv is active.

### Bug 3: `is_v32` AttributeError in GlmMoeDsaForCausalLM

`GlmMoeDsaForCausalLM` (line 1726) inherits from `DeepseekV2ForCausalLM` (line 1365).
`is_v32 = False` was only set in child class `__init__` methods but NOT in the
parent `DeepseekV2ForCausalLM.__init__`. During `load_weights()`, parent code
accesses `self.is_v32` before any child class sets it:

```
AttributeError: 'GlmMoeDsaForCausalLM' object has no attribute 'is_v32'
```

**Fix in `/data/vllm-ds4-sm120/vllm/model_executor/models/deepseek_v2.py`:**
Add `self.is_v32 = False` at 3 locations:

| Location | Class | Line (approx) |
|----------|-------|---------------|
| Child module | `GlmMoeDecoderLayer` | ~990 |
| Child module | `DeepseekV2MoE` | ~1221 |
| **Parent class** | `DeepseekV2ForCausalLM` | **~1384 (NEW â€” was missing)** |

Also fix the sparse indexer bug in `load_weights()`:
```python
# WRONG (unquoted â€” Python treats as undefined variable):
if not self.is_v32 and indexer in name:
    continue
# CORRECT (string literal):
if not self.is_v32 and "indexer" in name:
    continue
```

### Bug 4: Blackwell SM 12.0 Shared Memory Limit â€” Triton MLA OOF

Blackwell SM 12.0 hardware limit: **99 KB (101,376 bytes)** per thread block.
Triton's MLA decode kernel uses `num_stages=2` â†’ ~100 KB required â†’ exceeds limit:

```
triton.runtime.errors.OutOfResource: out of resource: shared memory,
Required: 102400, Hardware limit: 101376
```

**Fix in `/data/vllm-ds4-sm120/vllm/v1/attention/ops/triton_decode_attention.py`:**
Change all `num_stages=2` â†’ `num_stages=1` (two locations, lines ~493 and ~498):

Sync to worker:
```bash
base64 /data/vllm-ds4-sm120/vllm/v1/attention/ops/triton_decode_attention.py \
  | ssh jianliu@10.10.70.98 "base64 -d > /data/vllm-ds4-sm120/vllm/v1/attention/ops/triton_decode_attention.py \
    && python3 -c \"import py_compile; py_compile.compile('/data/vllm-ds4-sm120/vllm/v1/attention/ops/triton_decode_attention.py')\" \
    && echo synced"
```

### Bug 5: XQA MLA Backend Incompatibility

```
RuntimeError: XQA MLA only supports 128 query heads (head_group_ratio=128), got 8 query heads
```
Fix: startup script must use `--attention-config.backend=TRITON_MLA --enforce-eager`.

### Bug 6: Stale VLLM Workers Holding GPU Memory (~47 GB/GPU)

After `pkill -9 vllm` on the master, old `VLLM::Worker` subprocesses survive
independently (reparented to PID 1) holding ~47 GB/GPU VRAM. NOT visible in
`ps aux` but VISIBLE in `nvidia-smi --query-compute-apps`:

```
ValueError: Free memory on device cuda:5 (47.61/94.97 GiB) ... less than desired GPU memory utilization (0.8, 75.98 GiB)
```

**Diagnostic:** `nvidia-smi --query-compute-apps=pid,process_name,used_memory --format=csv`
**Fix:** Explicitly `kill -9` each PID from nvidia-smi output, then verify memory returns to ~17 MiB.

### Bug 7: TCPStore Port 29505 EADDRINUSE

**Fix:** `fuser -k 29505/tcp`

| Error | Root Cause | Fix |
